## 檔案命名規則

### 基本原則

- **全部使用小寫字母**
- **使用連字號 `-` 分隔單詞**（不使用底線或空格）
- **包含日期前綴**（格式：`YYYY-MM-DD`）
- **簡潔且描述性強**

### 命名格式

```
YYYY-MM-DD-主題關鍵詞.md
```

### 範例

✅ **正確範例：**
- `2025-11-20-claude-writing-guide.md`
- `2025-11-20-javascript-async-await.md`
- `2025-11-20-productivity-tips.md`

❌ **錯誤範例：**
- `Claude Writing Guide.md`（包含空格和大寫）
- `2025_11_20_guide.md`（使用底線）
- `writing-guide.md`（缺少日期）
- `2025-11-20-Guide.md`（包含大寫字母）

### 特殊情況

- **系列文章：** `2025-11-20-series-name-part-01.md`
- **筆記：** `2025-11-20-note-topic.md`
- **教學：** `2025-11-20-tutorial-topic.md`

## Heading 規則

### 層級結構

```markdown
# H1 - 文章標題（每篇文章只能有一個）

## H2 - 主要章節

### H3 - 子章節

#### H4 - 細分主題

##### H5 - 更細的分類（謹慎使用）

###### H6 - 最細分類（儘量避免）
```

### 使用原則

1. **H1 標題：**
   - 每篇文章只有一個 H1
   - H1 是文章的主標題
   - 應該清晰描述文章主題

2. **H2 標題：**
   - 用於劃分文章的主要章節
   - 應該邏輯清晰，層次分明

3. **H3-H6 標題：**
   - 用於更細的內容劃分
   - 避免層級過深（建議不超過 H4）

### 標題格式規範

```markdown
✅ 正確格式：
# 文章標題

## 第一章節

### 子主題

❌ 錯誤格式：
#文章標題（# 後缺少空格）
##第一章節（## 後缺少空格）
```

### 標題命名建議

- 使用清晰、描述性的標題
- 避免過長的標題（建議不超過 50 字元）
- 使用動詞開頭更具行動性（如「理解」、「掌握」、「實現」）
- 保持同級標題的語法一致性

### 標準文章結構

```markdown

- **建立日期：** YYYY-MM-DD
- **更新日期：** YYYY-MM-DD
- **標籤：** #tag1 #tag2 #tag3
- **狀態：** 草稿/已發布

## 目錄

（對於長文章，可選）

## 引言

文章的開頭，說明寫作目的和背景。

## 主要內容

### 章節 1

內容...

### 章節 2

內容...

## 總結

總結文章要點。

## 參考資料

- 參考連結 1
- 參考連結 2

## 相關文章

- [相關文章 1](連結)
```

#### 1. **技術文章**
- 問題描述
- 解決方案
- 程式碼範例
- 最佳實踐
- 總結

#### 2. **教學**
- 前置知識
- 步驟詳解

#### 3. **筆記**
- 主題

產業/技術/市場深度研究報告，目標達到市調或顧問公司水準。

**與公司研究的差異：**
- 涵蓋整個產業鏈上中下游多家公司

- **產業分析**：如電子貨架標籤、AI 晶片

4. 競爭格局分析
5. 應用場景與案例
6. 商業模式分析
7. 未來展望

**品質標準：**
- 數據支撐（市場規模、成長率、佔比等）
- 完整的參考資料
- 多家公司分析
- 技術深度解析

## 內容規範

### 語言風格

- **清晰簡潔：** 避免冗長複雜的句子
- **主動語態：** 優先使用主動語態
- **一致性：** 保持術語和表達方式的一致
- **讀者導向：** 考慮讀者的背景和需求

### 用語規範

#### 使用台灣繁體中文用語

避免使用中國慣用詞彙，採用台灣繁體中文的常見用語：

| 避免使用（中國用語） | 建議使用（台灣用語） |
|---------------------|---------------------|
| 文檔 | 文件 |
| 代碼 | 程式碼 |
| 視頻 | 影片 |
| 音頻 | 音訊 |
| 軟件 | 軟體 |
| 硬件 | 硬體 |
| 數據庫 | 資料庫 |
| 服務器 | 伺服器 |
| 網絡 | 網路 |
| 鏈接 | 連結 |
| 用戶 | 使用者 |
| 信息 | 資訊 |
| 默認 | 預設 |
| 光標 | 游標 |
| 優化 | 最佳化 |
| 交互 | 互動 |
| 字節 | 位元組 |
| 調用 | 呼叫 |

#### 避免特定句型

避免使用「這不是 X，而是 Y」的對比句型，此句型過度使用會讓文章顯得重複和單調。建議改用更自然的表達方式：

**❌ 避免的例子：**
- 「這不是私下的商業角力，而是公開的攻擊。」
- 「這不是陰謀論，而是商業競爭的常態。」
- 「這不是技術問題，而是感知問題。」

**✅ 五種改寫方式：**

**1. 直述句**（直接陳述 Y，省略對 X 的否定）
```markdown
原句：「這不是私下的商業角力，而是公開的攻擊。」
改寫：「這是公開的、實名的、針對性的攻擊。」
```

**2. 轉折句**（用轉折詞連接）
```markdown
原句：「這不是陰謀論，而是商業競爭的常態。」
改寫：「這看似陰謀論，實際上是商業競爭的常態。」
```

**3. 因果邏輯**（強調原因或結果）
```markdown
原句：「這不是技術問題，而是感知問題。」
改寫：「感知問題需要 CEO 親自處理，而非交給公關部門。」
```

**4. 讓步轉折**（雖然⋯⋯卻⋯⋯）
```markdown
原句：「這不是理性的商業決策，而是現實的風險管理。」
改寫：「雖然不符合理性的商業邏輯，卻是現實的風險管理。」
```

**5. 強調主語**（重新組織句子結構）
```markdown
原句：「這不是 Airwallex 的錯，但卻是它必須承擔的代價。」
改寫：「Airwallex 必須承擔這個代價，即使這不是他們的錯。」
```

**其他替代句型模板：**
- 「與其說是 X，不如說是 Y」
- 「表面上是 X，實則是 Y」
- 「X 只是表象，Y 才是核心」
- 「超越了 X 的層次，進入 Y 的境界」

### 標點符號規範

依據[教育部《重訂標點符號手冊》修訂版](https://language.moe.gov.tw/001/upload/files/site_content/m0001/hau/haushou.htm)，中文寫作應使用全形標點符號。

#### 基本原則

| 標點符號 | 正確用法 | 錯誤用法 | 說明 |
|---------|---------|---------|------|
| 逗號 | ， | , | 全形逗號 |
| 句號 | 。 | . | 全形句號 |
| 冒號 | ： | : | 全形冒號 |
| 分號 | ； | ; | 全形分號 |
| 問號 | ？ | ? | 全形問號 |
| 驚嘆號 | ！ | ! | 全形驚嘆號 |
| 頓號 | 、 | - | 用於並列詞語 |
| 引號 | 「」『』 | "" '' | 台灣繁體中文引號 |
| 破折號 | —— | -- | 佔兩格，中間不斷開 |
| 刪節號 | …… | ... | 佔兩格，六個點 |

#### 避免過度使用冒號

冒號用於「提示下文」或「總結上文」，不應過度使用。

```markdown
❌ 過度使用冒號：
「公司優勢：技術領先：擁有專利：超過 100 項。」

✅ 改寫方式：
「公司優勢在於技術領先，擁有超過 100 項專利。」

❌ 不必要的冒號：
「他說：他明天會來。」

✅ 改寫方式：
「他說他明天會來。」
「他說：『我明天會來。』」（直接引語才用冒號加引號）
```

**冒號使用時機：**
1. 提示直接引語：他說：「這是正確的。」
2. 引出列舉項目：主要產品包括：A、B、C 三項。
3. 總結上文：簡言之：這是一個好機會。
4. 標題與說明：注意：以下內容很重要。

#### 避免過度使用破折號

破折號用於「語意轉折」、「補充說明」或「聲音延續」，過度使用會使文章顯得零碎。

```markdown
❌ 過度使用破折號：
「這家公司——成立於 2020 年——專注於 AI 領域——已經獲得多輪融資——估值超過 10 億美元。」

✅ 改寫方式：
「這家公司成立於 2020 年，專注於 AI 領域，已獲得多輪融資，估值超過 10 億美元。」

✅ 適當使用破折號：
「這家公司專注於 AI 領域——特別是自然語言處理——已成為業界領導者。」
```

**破折號使用時機：**
1. 語意轉折：我本想去，但是——算了，不說了。
2. 補充說明：台北——台灣的首都——是一座現代化城市。
3. 聲音延續：「嗚——」火車鳴笛而過。
4. 時空起止：2020——2025 年

**建議：** 每段文字中破折號不超過 1-2 處，優先考慮使用逗號、括號或改寫句子。

#### 中英文混排規範

```markdown
✅ 正確：
「Claude 是一款 AI 助手。」（中英文之間加空格）
「使用 API 進行開發。」

❌ 錯誤：
「Claude是一款AI助手。」（缺少空格）
```

### 商業寫作風格

以下是提升文章品質的商業寫作技巧，特別適用於深度分析類文章。

#### 敘事性

- 使用場景描寫開頭，避免直接進入論述
- 加入具體時間、地點、人物，增加臨場感
- 用問句引導讀者思考，創造互動感

```markdown
❌ 避免：
「AI 產業正在快速發展，以下分析其趨勢。」

✅ 建議：
「2025 年 12 月 1 日，矽谷投資圈炸鍋了。一份來自華爾街的報告指出⋯⋯」
```

#### 避免過度使用列點

- 將項目符號改寫成流暢的段落
- 用連接詞串聯想法（首先、其次、此外、最後）
- 保持敘事的連貫性，讓讀者順暢閱讀

```markdown
❌ 過度列點：
公司優勢：
- 技術領先
- 資金充足
- 團隊經驗豐富

✅ 改寫成段落：
「這家公司的優勢首先體現在技術層面，其專利數量領先競爭對手。
此外，在 B 輪融資後資金充足，加上創始團隊在業界的豐富經驗，
使其具備長期競爭力。」
```

#### 金句與比喻

適當使用隱喻和比喻，創造易記的金句：

- 「CFIUS 的達摩克利斯之劍」
- 「疑慮比事實更有傳播力」
- 「這是一場沒有硝煙的戰爭」

**注意：** 金句應自然融入文章，避免為了金句而金句。

#### 段落過渡

每個段落之間要有自然過渡，避免突兀的主題轉換：

```markdown
❌ 突兀轉換：
「⋯⋯公司營收成長 50%。
接下來談談競爭對手。」

✅ 自然過渡：
「⋯⋯公司營收成長 50%。
然而，這樣的成績並非沒有挑戰。就在同一時期，競爭對手也沒有閒著。」
```

#### 商業寫作檢查清單

- [ ] 文章開頭具有敘事性（有場景、時間、人物）
- [ ] 使用了適當的問句引導讀者思考
- [ ] 列點適度，重要論述以段落呈現
- [ ] 段落之間有流暢的過渡句
- [ ] 有 1-2 個適當的金句或比喻
- [ ] 整體閱讀流暢，不會感覺零碎

## 發布到 Multivac42

### 架構原則：單一來源 (Single Source of Truth)

Pensieve 是所有內容的**唯一編輯來源**，Multivac42 只是發布的「快照」。

```
Pensieve (編輯來源)          Multivac42 (公開網站)
─────────────────────        ─────────────────────
修改文章
    │
    ▼ 執行發布腳本
    │
    └──────────────────────> 覆蓋舊版本
```

**重要原則：**
- ✅ 所有修改都在 Pensieve 進行
- ✅ 修改後重新執行發布腳本即可更新
- ❌ 不要直接編輯 Multivac42 中的文章（會被覆蓋）

### Frontmatter 欄位

發布到 Multivac42 的文章需要包含以下 frontmatter：

```yaml
---
title: "文章標題"
description: "文章摘要（1-2 句話）"
date: 2025-12-03
author: "Clement Tang"
```

| 欄位 | 必填 | 說明 |
|------|------|------|
| `title` | ✅ | 文章標題 |
| `description` | ✅ | 文章摘要，用於 SEO 和預覽 |
| `date` | ✅ | 發布日期（YYYY-MM-DD） |
| `author` | ✅ | 作者名稱 |
| `tags` | ✅ | 標籤陣列 |
| `category` | ✅ | 分類（決定 M42 目標目錄，見下表） |
| `status` | ✅ | 文章狀態（draft / in-progress / published / archived） |
| `lastModified` | ⚪ | 最後修改日期（可選，腳本會自動補充） |

**Status 欄位說明：**
- `draft` - 草稿，尚未完成
- `in-progress` - 撰寫中
- `published` - 已發布（觸發發布到 Multivac42）
- `archived` - 已歸檔，不再維護

**Category 對應的 M42 目標目錄：**

| Category | M42 目標目錄 | 結構 |
|----------|-------------|------|
| `articles` | `docs/articles/` | 平面結構 |
| `company-research` | `docs/company-research/{公司名}/` | 依公司分類 |
| `topic-research` | `docs/topic-research/` | 平面結構 |

> **注意：** Pensieve 內文章可放在任何目錄，發布腳本會根據 `category` 欄位決定 M42 的目標位置。

### 發布流程

1. **標記文章**：在 frontmatter 設定 `status: published`
2. **執行腳本**：
   ```bash
   cd ~/pensieve
   node scripts/publish-to-multivac.js
   ```
3. **提交 Multivac42**：
   ```bash
   cd ~/multivac42
   git add -A
   git commit -m "發布新文章：文章標題"
   git push
   ```

### 格式轉換

發布腳本會自動進行以下轉換：

| 原始格式 | 發布格式 |
|----------|----------|
| `## 元資料` 區塊 | 移除（由 Byline 組件取代） |
| `*最後更新：...*` | 移除 |
| 缺少 `lastModified` | 自動補充檔案修改時間 |

### 同步狀態檢查

發布腳本會提示：
- 「有 N 篇文章已修改但未重新發布」
- 「有 N 篇新文章標記為 status: published」
