## 檔案命名規則

### 基本原則

- **全部使用小寫字母**
- **使用連字號 `-` 分隔單詞**（不使用底線或空格）
- **包含日期前綴**（格式：`YYYY-MM-DD`）
- **簡潔且描述性強**

### 命名格式

```
YYYY-MM-DD-主題關鍵詞.md
```

### 範例

✅ **正確範例：**
- `2025-11-20-claude-writing-guide.md`
- `2025-11-20-javascript-async-await.md`
- `2025-11-20-productivity-tips.md`

❌ **錯誤範例：**
- `Claude Writing Guide.md`（包含空格和大寫）
- `2025_11_20_guide.md`（使用底線）
- `writing-guide.md`（缺少日期）
- `2025-11-20-Guide.md`（包含大寫字母）

### 特殊情況

- **系列文章：** `2025-11-20-series-name-part-01.md`
- **筆記：** `2025-11-20-note-topic.md`
- **教學：** `2025-11-20-tutorial-topic.md`

## Heading 規則

### 層級結構

```markdown
# H1 - 文章標題（每篇文章只能有一個）

## H2 - 主要章節

### H3 - 子章節

#### H4 - 細分主題

##### H5 - 更細的分類（謹慎使用）

###### H6 - 最細分類（儘量避免）
```

### 使用原則

1. **H1 標題：**
   - 每篇文章只有一個 H1
   - H1 是文章的主標題
   - 應該清晰描述文章主題

2. **H2 標題：**
   - 用於劃分文章的主要章節
   - 應該邏輯清晰，層次分明

3. **H3-H6 標題：**
   - 用於更細的內容劃分
   - 避免層級過深（建議不超過 H4）

### 標題格式規範

```markdown
✅ 正確格式：
# 文章標題

## 第一章節

### 子主題

❌ 錯誤格式：
#文章標題（# 後缺少空格）
##第一章節（## 後缺少空格）
```

### 標題命名建議

- 使用清晰、描述性的標題
- 避免過長的標題（建議不超過 50 字元）
- 使用動詞開頭更具行動性（如「理解」、「掌握」、「實現」）
- 保持同級標題的語法一致性

- **標籤：** #tag1 #tag2 #tag3

## 內容規範

### 語言風格

- **清晰簡潔：** 避免冗長複雜的句子
- **主動語態：** 優先使用主動語態
- **一致性：** 保持術語和表達方式的一致
- **讀者導向：** 考慮讀者的背景和需求

### 用語規範

#### 使用台灣繁體中文用語

避免使用中國慣用詞彙，採用台灣繁體中文的常見用語：

| 避免使用（中國用語） | 建議使用（台灣用語） |
|---------------------|---------------------|
| 文檔 | 文件 |
| 代碼 | 程式碼 |
| 視頻 | 影片 |
| 音頻 | 音訊 |
| 軟件 | 軟體 |
| 硬件 | 硬體 |
| 數據庫 | 資料庫 |
| 服務器 | 伺服器 |
| 網絡 | 網路 |
| 鏈接 | 連結 |
| 用戶 | 使用者 |
| 信息 | 資訊 |
| 默認 | 預設 |
| 光標 | 游標 |
| 優化 | 最佳化 |
| 交互 | 互動 |
| 字節 | 位元組 |
| 調用 | 呼叫 |

#### 避免特定句型

避免使用「這不是 X，而是 Y」的對比句型，此句型在中文寫作中顯得生硬。建議改用更自然的表達方式：

```markdown
❌ 避免：
「這不是一個創業成功的故事，而是一場正在上演的革命。」

✅ 建議改寫：
「這個故事的意義遠超創業成功本身——它代表著一場正在上演的革命。」
「與其說是創業成功，更準確地說，這是一場正在上演的革命。」
「表面上看是創業成功，實則是一場正在上演的革命。」
```

其他替代句型參考：
- 「與其說是 X，不如說是 Y」
- 「表面上是 X，實則是 Y」
- 「X 只是表象，Y 才是核心」
- 「超越了 X 的層次，進入 Y 的境界」
- 直接陳述 Y，省略對 X 的否定

### 段落規範

- 每個段落專注一個主題
- 段落之間空一行
- 避免過長的段落（建議 3-5 句）

### 列表使用

```markdown
✅ 有序列表（步驟、優先順序）：
1. 第一步
2. 第二步
3. 第三步

✅ 無序列表（並列要點）：
- 要點 1
- 要點 2
- 要點 3

✅ 任務列表：
- [ ] 待完成任務
- [x] 已完成任務
```

## Markdown 格式規範

### 程式碼區塊

````markdown
```語言名稱
程式碼內容
```

範例：

```javascript
const greeting = "Hello, World!";
console.log(greeting);
```
````

### 行內程式碼

使用反引號包裹：`` `程式碼` ``

範例：使用 `console.log()` 輸出資訊。

### 連結

```markdown
[連結文字](URL)

範例：[Claude AI](https://claude.ai)
```

### 圖片

```markdown
![圖片描述](圖片URL)

範例：![範例圖片](./images/example.png)
```

### 引用

```markdown
> 這是一段引用文字
> 可以多行
```

### 強調

```markdown
**粗體文字**
*斜體文字*
~~刪除線~~
```

### 表格

```markdown
| 欄1 | 欄2 | 欄3 |
|-----|-----|-----|
| 資料1 | 資料2 | 資料3 |
| 資料4 | 資料5 | 資料6 |
```

### 水平線

```markdown

```

## 元資料規範

### 文章元資料

每篇文章應在開頭包含以下元資料：

```markdown
## 元資料

- **建立日期：** 2025-11-20
- **更新日期：** 2025-11-20
- **作者：** [你的名字]
- **標籤：** #技術 #寫作 #Claude
- **狀態：** 草稿
- **字數：** 約 2000 字
```

### 標籤系統

使用標籤分類文章：

- **技術類：** `#程式設計` `#網頁開發` `#資料科學`
- **工具類：** `#Claude` `#Git` `#VSCode`
- **方法類：** `#教學` `#指南` `#最佳實踐`
- **主題類：** `#生產力` `#學習` `#思考`

### 狀態標記

- **草稿：** 正在寫作中
- **審閱中：** 等待審閱
- **已發布：** 完成並發布
- **歸檔：** 已過時的內容

## 發布到 Multivac42

### 架構原則：單一來源 (Single Source of Truth)

Pensieve 是所有內容的**唯一編輯來源**，Multivac42 只是發布的「快照」。

```
Pensieve (編輯來源)          Multivac42 (公開網站)
─────────────────────        ─────────────────────
修改文章
    │
    ▼ 執行發布腳本
    │
    └──────────────────────> 覆蓋舊版本
```

**重要原則：**
- ✅ 所有修改都在 Pensieve 進行
- ✅ 修改後重新執行發布腳本即可更新
- ❌ 不要直接編輯 Multivac42 中的文章（會被覆蓋）

### Frontmatter 欄位

發布到 Multivac42 的文章需要包含以下 frontmatter：

```yaml
---
title: "文章標題"
description: "文章摘要（1-2 句話）"
date: 2025-12-03
author: "Clement Tang"
tags: ["標籤1", "標籤2"]
category: articles  # 或 company-research、topic-research
publish: true       # 設為 true 才會被發布
---
```

| 欄位 | 必填 | 說明 |
|------|------|------|
| `title` | ✅ | 文章標題 |
| `description` | ✅ | 文章摘要，用於 SEO 和預覽 |
| `date` | ✅ | 發布日期（YYYY-MM-DD） |
| `author` | ✅ | 作者名稱 |
| `tags` | ✅ | 標籤陣列 |
| `category` | ✅ | 分類：articles、company-research、topic-research |
| `publish` | ✅ | 是否發布到 Multivac42 |
| `lastModified` | ⚪ | 最後修改日期（可選，腳本會自動補充） |

### 發布流程

1. **標記文章**：在 frontmatter 加入 `publish: true`
2. **執行腳本**：
   ```bash
   cd ~/pensieve
   node scripts/publish-to-multivac.js
   ```
3. **提交 Multivac42**：
   ```bash
   cd ~/multivac42
   git add -A
   git commit -m "發布新文章：文章標題"
   git push
   ```

### 格式轉換

發布腳本會自動進行以下轉換：

| 原始格式 | 發布格式 |
|----------|----------|
| `## 元資料` 區塊 | 移除（由 Byline 組件取代） |
| `*最後更新：...*` | 移除 |
| 缺少 `lastModified` | 自動補充檔案修改時間 |

### 同步狀態檢查

發布腳本會提示：
- 「有 N 篇文章已修改但未重新發布」
- 「有 N 篇新文章標記為 publish: true」
