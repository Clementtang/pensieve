## 分支命名規則

### 命名格式

```
<類型>/<簡短描述>-<會話ID>
```

### 類型前綴

- `claude/` - 由 Claude AI 協助的分支
- `feature/` - 新功能開發
- `fix/` - 問題修復
- `docs/` - 文件更新
- `refactor/` - 程式碼重構

### 範例

- `claude/luckin-coffee-research-01KDkVyaMo84sk1bgnp9BRuX`
- `feature/add-new-template`
- `docs/update-writing-guide`

## 提交訊息規範

### 格式原則

- 使用清晰、描述性的訊息
- 第一行簡短概述（建議 50 字元以內）
- 如需詳細說明，第二行空白，第三行開始寫詳細描述

### 提交訊息範例

✅ **好的提交訊息：**
```
新增企業研究：瑞幸咖啡深度分析

- 新增 2.5 萬字企業研究報告
- 包含財務造假醜聞與翻轉故事
- 分析競爭優勢與市場策略
```

```
深度更新：新增瑞幸vs星巴克競爭分析與台灣市場評估

- 新增五大核心競爭優勢分析
- 新增台灣市場進入可行性評估
- 補充學術研究與產業數據
```

```
新增文章：瑞幸咖啡進軍台灣市場深度分析

- 2,800 字社群媒體友善文章
- 聚焦台灣市場挑戰與機會
- 商業周刊風格寫作
```

❌ **不好的提交訊息：**
```
更新檔案
修改內容
完成
fix
```

## Pull Request 規範

### PR 標題格式

使用簡潔、清晰的標題，直接說明本次 PR 的主要內容：

**格式範例：**
```
新增<類型>：<簡短描述>
```

**類型說明：**
- `新增文章` - 新增正式文章
- `新增企業研究` - 新增企業研究報告
- `新增筆記` - 新增筆記內容
- `新增模板` - 新增文章模板
- `更新文章` - 更新現有文章
- `修正` - 修正錯誤或問題
- `重構` - 程式碼或結構重構

**PR 標題範例：**

✅ **好的 PR 標題：**
```
新增企業研究：瑞幸咖啡深度分析與台灣市場評估
新增文章：Claude Opus 4.5 發布分析
更新文章：Amazon Leo Ultra 補充市場數據
修正：修復文章中的錯別字和格式問題
新增模板：企業研究報告模板
```

❌ **不好的 PR 標題：**
```
Update files
新增內容
修改
完成寫作
Add article
```

### PR 描述格式

PR 描述應該清晰說明本次變更的內容和範圍。

**標準格式：**

```markdown
## 新增內容

1. **<內容類型>**（字數或規模）
   - 主要內容摘要點 1
   - 主要內容摘要點 2
   - 主要內容摘要點 3

2. **<其他內容類型>**（字數或規模）
   - 主要內容摘要點 1
   - 主要內容摘要點 2

## 檔案清單

- `路徑/檔案名稱.md`
- `路徑/檔案名稱.md`

## 備註（可選）

- 特殊說明或注意事項
- 相關 issue 連結
```

**PR 描述範例：**

```markdown
## 新增內容

1. **企業研究報告**（2.5萬字）
   - 瑞幸咖啡從財務造假到48億美元營收的翻轉故事
   - 深度解析：如何在中國打敗星巴克的五大競爭優勢
   - 台灣市場評估與成功機率分析

2. **深度文章**（2,800字）
   - 聚焦瑞幸12月進軍台灣市場
   - 分析便利商店咖啡的競爭挑戰
   - 三種情境的成功機率評估

## 檔案清單

- `docs/company-research/2025-11-28-luckin-coffee-research.md`
- `docs/articles/2025-11-28-luckin-coffee-taiwan-analysis.md`
```

### PR 檢查清單

提交 PR 前，請確認：

- [ ] 遵循檔案命名規則（參考 [WRITING_GUIDE.md](./WRITING_GUIDE.md)）
- [ ] 文章包含完整的元資料
- [ ] Markdown 格式正確
- [ ] 連結都可用
- [ ] 沒有明顯的錯別字或語法錯誤
- [ ] PR 標題和描述清晰明確
- [ ] 已將變更推送到正確的分支

## 程式碼審查

### 審查重點

1. **內容品質**
   - 資訊是否準確
   - 邏輯是否清晰
   - 表達是否流暢

2. **格式規範**
   - 是否符合 WRITING_GUIDE.md
   - Markdown 格式是否正確
   - 檔案命名是否規範

3. **元資料完整性**
   - 是否包含必要的元資料
   - 標籤是否適當
   - 日期是否正確

### 審查回饋建議

- 提供具體、建設性的意見
- 指出具體的改進點，並說明原因
- 如有範例更好

## 協作流程

### 1. 建立分支

```bash
git checkout -b claude/your-topic-description-sessionID
```

### 2. 進行變更

- 遵循寫作規範進行內容創作
- 定期提交變更（使用清晰的提交訊息）

```bash
git add .
git commit -m "新增文章：主題描述"
```

### 3. 推送到遠端

```bash
git push -u origin claude/your-topic-description-sessionID
```

### 4. 建立 Pull Request

- 使用規範的 PR 標題
- 填寫完整的 PR 描述
- 如有需要，指定審查者

### 5. 回應審查意見

- 及時回應審查意見
- 進行必要的修改
- 更新 PR

### 6. 合併

- 等待審查通過
- 確認所有檢查通過
- 合併到主分支
